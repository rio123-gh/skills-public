# 現場向け：探索的データ分析プロンプト集

AIエージェントに一度で完璧な指示を出す必要はありません。  
良いプロンプトとは、**「AIに最初のグラフを描かせ、そこから対話をスタートさせるためのパス」**です。

---

## 1. 分析のスタート（視覚化の依頼）

最初は難しい分析手法を指定せず、「見たいもの」と「現場の違和感」だけを伝えて、視覚的な手がかりを引き出します。

### ❌ 悪いプロンプト

```
Cラインの歩留まりデータの重回帰分析をして、不良要因を特定してください。
```

**理由**: 分析手法（重回帰分析）を決め打ちしてしまうと、AIが他の最適な手法（決定木など）を提案する余地を奪ってしまいます。また、いきなり要因を特定しようとすると、結果の解釈がブラックボックス化しやすくなります。

### ✅ 良いプロンプト

```
先週のCラインの生産データで、歩留まりが落ちている気がします。
まずは全体の温度と圧力のトレンドグラフを出してくれますか？
```

**理由**: 「グラフを出す」という具体的な最初のアクションを指示しつつ、「歩留まりが落ちている気がする」という現場のコンテキストをAIに共有しています。ここからマルチターンの対話が始まります。

---

## 2. 異常値の深掘り（XAIスキルの活用）

グラフを見て違和感を見つけたら、AIの「要因特定機能（SHAP等）」を呼び出して、現場の直感と照らし合わせます。

### ❌ 悪いプロンプト

```
このデータの特徴量重要度を計算して。
```

**理由**: データサイエンスの専門用語を使ってしまっており、本来の業務目的が見えません。また、全体の重要度だけを見ても、特定の異常事象の解決には繋がらないことが多いです。

### ✅ 良いプロンプト

```
水曜日の深夜2時頃に急激な圧力低下がありますね。
この特定の時間帯の低下に、一番悪影響を与えていたセンサー値は何ですか？
内訳を教えて。
```

**理由**: ピンポイントな事象（水曜深夜の低下）に対する「なぜ？」を問いかけています。これにより、AIは特定のデータ点に対する詳細な要因分析ツール（Waterfall Plotなど）を正確に実行できます。

---

## 3. コンテキストの更新（除外と条件指定）

現場の「当たり前」をAIに教えながら、分析対象のデータを洗練させていきます。

### ❌ 悪いプロンプト

```
statusカラムが 'cleaning' と 'maintenance' の行を削除した上で、
再度同じ分析コードを実行して出力してください。
```

**理由**: データベース言語のような指示になっており、作成のハードルが高いです。

### ✅ 良いプロンプト

```
さっきのノイズ部分は洗浄工程なので分析から除外してください。
その上で、残りのデータを使って、午前と午後のシフトでセンサー値の分布に違いがないか比較してみて。
```

**理由**: 前の会話の文脈を引き継ぎつつ、自然言語でデータのフィルタリングと「セグメント比較」の指示を出せています。

---

## 4. 時間的遅れ（ラグ）の確認

製造現場特有の「波及効果」や「タイムラグ」を調べるための指示です。

### ❌ 悪いプロンプト

```
バルブ開度と製品温度の交差相関関数（相互相関）をプロットして。
```

**理由**: 専門用語を使うことで、他の現場担当者がプロンプトを再利用しにくくなります。

### ✅ 良いプロンプト

```
ヒーターの出力を上げてから、実際の製品温度が上がり始めるまでにタイムラグがあるはずです。
何分遅れで影響が出ているか、ズレを可視化して確認できますか？
```

**理由**: 現場の物理的な現象（ヒーターと温度のズレ）をそのまま言葉にしています。裏側でAIエージェントが適切な時系列分析ツール（ラグ相関分析など）を選択しやすくなります。

---

## 5. 仮説のシミュレーション

原因の仮説が立ってきたら、それが全体にどう波及するかをAIにシミュレーションさせます。

### ❌ 悪いプロンプト

```
この変数を用いたVARモデルを構築して。
```

### ✅ 良いプロンプト

```
もし圧力センサーAの数値が今の状態から急上昇した場合、
その後数時間で他の設備にどういう影響が波及しそうか、シミュレーションして見せて。
```

**理由**: 「もし〜だったらどうなる？」という現場目線の問いかけ（What-if分析）になっています。
