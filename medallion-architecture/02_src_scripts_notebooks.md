srcとscriptsとnotebooksの使い分けについて教えてください。(チームデータ分析PJです)

チームでのデータ分析プロジェクト（しかもセンサー、ドキュメント、CSVなどが入り組む環境）において、コード管理のディレクトリをどう分けるかは、将来の「カオス（誰が書いたかわからない野良スクリプトの乱立）」を防ぐ上で極めて重要です。

src/、scripts/、notebooks/ は、「そのコードの目的」と「実行される環境（本番か、開発者の手元か）」によって厳密に使い分けるのがベストプラクティスです。

それぞれの役割と使い分けの基準を解説します。

1. src/（Source）: プロジェクトの「心臓部・部品庫」
目的: 再利用可能で、テスト可能で、システムとして安定稼働させるための汎用的な関数やクラス群（モジュール）を置く場所です。

特徴:

ここにあるファイル単体を直接実行する（python src/load_data.py のように叩く）ことは原則としてしません。他のスクリプトやノートブックから import して使われる「部品」です。

単体テスト（tests/）の対象になるのは、基本的にこの中身です。

入れるべき内容の例:

データベースへの接続クラス（src/db_connector.py）

センサーデータの異常値を除去するアルゴリズム（src/preprocessing/sensor_filters.py）

日報のPDFからテキストを抽出する関数（src/features/pdf_parser.py）

2. scripts/ : 自動化とシステム実行の「指示書」
目的: src/ にある部品を組み合わせて、一連のパイプライン（ETL）やバッチ処理を「頭からお尻まで」実行するためのファイルを置く場所です。

特徴:

直接実行されること（python scripts/01_run_bronze_to_silver.py）を前提としています。

コマンドライン引数（--date 2026-06-03 など）を受け取って動くように作ります。

AirflowやCron、GitHub Actionsなどのジョブ管理ツール（オーケストレータ）から呼び出されるのは、このディレクトリのファイルです。

入れるべき内容の例:

BronzeからSilverへの変換バッチ（scripts/etl/bronze_to_silver_alarms.py）

毎朝実行する集計レポート生成処理（scripts/jobs/daily_aggregation.py）

データベースのテーブルを初期化するツール（scripts/setup/init_db.py）

3. notebooks/ : 探索と実験の「作業机」
目的: データの可視化、一時的な集計、機械学習モデルの実験など、トライ＆エラー（探索的データ分析：EDA）を行うためのJupyter Notebook（.ipynb）を置く場所です。

特徴:

本番環境（パイプライン）で実行してはいけません。

チームで共有する際は、結果が残るため便利ですが、Gitでコンフリクト（競合）しやすいため、ファイル名に個人の名前や日付を入れる運用（例: 01_takashi_eda_alarms.ipynb）が推奨されます。

ここで書いて「うまくいったロジック」は、そのままNotebookに残すのではなく、関数化して src/ に移し、scripts/ から呼び出すようにリファクタリングするのが黄金ルールです。

入れるべき内容の例:

「新しいセンサーのデータ形式をとりあえず見てみる」ためのノート（notebooks/eda/01_explore_new_sensor.ipynb）

モデルのパラメータチューニングの記録（notebooks/experiments/02_model_tuning.ipynb）

使い分けのフロー（チーム開発の黄金パターン）
実際のプロジェクトでは、コードはこの3つのディレクトリを以下のようなサイクルで移動します。

【Notebooks】 まずはNotebookを開き、新しいCSVデータを読み込んで色々試す。グラフを描いてデータの性質を理解する。

【Notebooks → Src】 「このCSVの日付フォーマットを直す処理」は今後も毎回使いそうだと気付く。Notebook上で書いたその処理を関数化し、src/preprocessing/date_formatter.py に移動する。

【Src → Scripts】 その関数を import し、毎日新しいCSVが届くたびに自動で変換処理を走らせるためのファイルを scripts/etl/run_daily_transform.py として作成する。

「実験はNotebookで、部品はSrcに、本番実行はScriptsで」という役割分担をチーム内で明確に合意しておくことで、レビューがしやすく、保守性の高いデータ基盤を構築できます。
